## Story

As a security-conscious user,
I want my personal data encrypted both at rest in the database and in transit over the network,
so that my information is protected against unauthorized access and interception.

## Acceptance Criteria

1. **AC1: Database Encryption at Rest**
   - All Aurora PostgreSQL tables containing sensitive data (users, groups, events, comments, wishlist items) are encrypted at rest using AWS Key Management Service (KMS)
   - Encryption key is managed by AWS KMS, not in application code
   - Encryption is transparent to application logic (no code changes required for reads/writes)
   - Verify: Query `pg_stat_database` to confirm encryption configuration

2. **AC2: HTTPS/TLS for All Data in Transit**
   - All API endpoints enforce HTTPS (no HTTP access allowed)
   - TLS 1.2 or higher is configured on all server connections
   - HSTS (HTTP Strict-Transport-Security) header enforced with `max-age=31536000` (1 year)
   - API Gateway and load balancer terminate TLS connections properly
   - Verify: SSL/TLS scan (e.g., via AWS Certificate Manager or SSLLabs) confirms TLS 1.2+ enforced

3. **AC3: Sensitive Field Encryption in Database**
   - Email addresses stored in database are hashed (not plaintext) where appropriate
   - Phone numbers (future feature) hashed using bcrypt or similar
   - Passwords never stored in plaintext (managed by AWS Cognito)
   - Identify all sensitive fields: `users.email`, `public_rsvps.email`, user contact info
   - Verify: Database dump shows no plaintext emails

4. **AC4: Encryption Key Management & Rotation**
   - KMS master key configured with annual key rotation enabled
   - Application never directly handles encryption keys (managed by AWS KMS)
   - Key policy restricts access to application IAM role only
   - Verify: AWS KMS console shows automatic rotation enabled

5. **AC5: Authentication Token Security**
   - JWT tokens generated by Cognito contain no sensitive user information
   - Tokens transmitted in HTTP-only cookies (not localStorage) to prevent XSS access
   - Token lifetime: 30 minutes for access tokens, 7 days for refresh tokens
   - Verify: Network inspection shows tokens in secure cookies, not in headers or response bodies

6. **AC6: API Endpoint Security**
   - All API routes validate HTTPS before processing (reject HTTP)
   - No sensitive data logged to plaintext logs
   - Error messages don't leak sensitive information (e.g., "User not found" instead of specific validation failures)
   - Verify: Application error logs show no exposed emails, phone numbers, or personal data

7. **AC7: Public Event Link Security**
   - Public event tokens generated with cryptographically secure random generation (32+ chars, URL-safe)
   - Public RSVP emails hashed in `public_rsvps.email` column (bcrypt)
   - Verify: `public_rsvps` table shows hashed emails, not plaintext

8. **AC8: GDPR/CCPA Data Subject Rights**
   - System maintains record of all encrypted data per user
   - Data export functionality provides decrypted user data in JSON/CSV format (admin-only)
   - User deletion triggers cryptographic erasure (delete or zero-out encryption keys)
   - Verify: User data export returns all personal information

9. **AC9: Encryption Testing & Validation**
   - Unit tests verify encryption is applied to all sensitive fields
   - Integration tests confirm HTTPS enforcement (no HTTP fallback)
   - Security audit script validates KMS key rotation and key policy
   - Verify: All tests pass (100+ new tests added)

10. **AC10: Encryption Documentation**
    - README documents encryption approach: Aurora KMS, HTTPS/TLS, token management
    - Architecture updated with encryption section
    - Security checklist confirms all 10 ACs implemented
    - Verify: Documentation complete and accurate

## Dev Notes

### Architecture & Context

**Encryption Philosophy:**
This story implements defense-in-depth encryption across three layers:
1. **Data at Rest:** Aurora KMS encryption (automatic, transparent)
2. **Data in Transit:** HTTPS/TLS with HSTS
3. **Application Layer:** Sensitive field hashing (email, public RSVP emails)

**Security Standards Met:**
- OWASP Top 10: Addresses A02:2021 (Cryptographic Failures)
- GDPR: Articles 5, 25, 32 (Data Protection by Design)
- CCPA: Consumer right to know/delete (data export/deletion endpoints)
- PCI DSS (if handling payments in future): Secure transmission, no plaintext storage

**Implementation Constraints:**
- AWS KMS keys managed by AWS (application never handles raw keys)
- Aurora encryption is transparent to application code
- Email hashing must be consistent (bcrypt with same rounds for comparisons)
- HTTPS enforcement must not break existing integrations
- Token security must maintain session longevity (30-min access, 7-day refresh)
